# RealtimeKit Webhook Log Explanation

This note explains the two Cloudflare log records seen for one real
RealtimeKit `meeting.ended` webhook delivered to the staging Worker.

Webhook endpoint:

```txt
POST https://cf-realtimekit-demo-staging.gaurkuber.workers.dev/api/realtimekit/webhook
```

## Short Version

The webhook worked.

RealtimeKit sent a `POST` request to the deployed Worker, the Worker accepted it
with HTTP `200`, and our application log captured the real webhook payload shape.

The two visible log lines are expected:

- One is Cloudflare platform telemetry for the HTTP request.
- One is our Worker application log for the parsed webhook payload.

The application log appears with an empty message because it is structured JSON
without a top-level human-readable `message` field.

## Log Record 1: Cloudflare Request Log

This log line looked like:

```txt
POST https://cf-realtimekit-demo-staging.gaurkuber.workers.dev/api/realtimekit/webhook
```

This is generated by Cloudflare Workers observability, not by our app code.

It proves:

- A real external caller reached the Worker.
- The method was `POST`.
- The route was `/api/realtimekit/webhook`.
- The response status was `200`.
- The request was handled by `cf-realtimekit-demo-staging`.
- Cloudflare assigned request/ray id `a01ef7402e68c9ab`.

Useful fields:

```json
{
  "method": "POST",
  "path": "/api/realtimekit/webhook",
  "responseStatus": 200,
  "requestId": "a01ef7402e68c9ab",
```

## Real Sender Evidence

The request headers strongly indicate this was the real RealtimeKit/Dyte webhook
sender.

Important headers observed:

```txt
dyte-signature: [redacted]
dyte-uuid: [uuid]
dyte-webhook-id: [uuid]
user-agent: axios/1.7.9
content-type: application/json
content-length: 417
```

What this means:

- `dyte-signature` means the sender provides a signature header.
- `dyte-uuid` appears to identify the webhook delivery attempt or event.
- `dyte-webhook-id` appears to identify the registered webhook.
- The source IP was an AWS IP in Ashburn, Virginia.
- The payload body was JSON.

Important security note: the signature value itself must not be copied into
docs, committed files, screenshots, or issue comments.

## Log Record 2: Worker Application Log

This log line came from our Worker route implementation:

```json
{
  "event": "realtimekit_webhook_received",
  "status": "accepted",
  "eventType": "meeting.ended"
}
```

This proves:

- Our route parsed the JSON body.
- The payload passed the webhook object schema.
- The route logged the event as accepted.
- The event type was `meeting.ended`.
- The same Cloudflare request id links it back to the platform request log.

The UI showed this as an empty message because our app log is structured data.
We did not include a top-level `message` string such as:

```json
{
  "message": "RealtimeKit webhook received"
}
```

That is not a functional problem. It just affects how the log row is displayed
in Cloudflare's log viewer.

The real inbound `meeting.ended` payload looked like this shape:

```json
{
  "event": "meeting.ended",
  "meeting": {
    "id": "bbb3d6ea-fa66-42b8-9cb7-fae1464908ff",
    "sessionId": "4f9aa050-823e-4970-ab37-756ca39b6842",
    "title": "mnm",
    "status": "LIVE",
    "createdAt": "2026-05-26T18:49:12.422Z",
    "startedAt": "2026-05-26T18:49:12.422Z",
    "endedAt": "2026-05-26T18:50:19.648Z",
    "organizedBy": {
      "id": "fc59a066-f019-4844-bf67-2a6ffb564e10",
      "name": "RTK_dde3fda5-8a83-4f2d-9776-d3da04de542a"
    }
  },
  "reason": "HOST_ENDED_MEETING"
}
```

The current app intentionally stores this as a bounded, redacted log snapshot
instead of persisting it in D1, R2, or files.

## What We Learned

The official docs did not clearly document the inbound webhook payload shape.
The real event gave us these implementation facts:

- The event type is in top-level `event`.
- Meeting id is nested at `meeting.id`.
- Session id is nested at `meeting.sessionId`.
- Meeting title is nested at `meeting.title`.
- End time is nested at `meeting.endedAt`.
- End reason is top-level `reason`.
- Organizer details are nested at `meeting.organizedBy`.
- `meeting.status` may still say `LIVE` for a `meeting.ended` event.

That last point matters: the app should treat top-level `event:
"meeting.ended"` and `meeting.endedAt` as stronger signals than
`meeting.status`.

## Why There Are Two Rows

For one incoming webhook request, Cloudflare can show at least two rows:

1. Platform request row:

   ```txt
   POST https://.../api/realtimekit/webhook
   ```

2. Application structured log row:

   ```json
   {
     "event": "realtimekit_webhook_received",
     "status": "accepted",
     "eventType": "meeting.ended"
   }
   ```

They are linked by the same request id / ray id:

```txt
a01ef7402e68c9ab
```

So this is not a duplicate webhook. It is two views of the same request:
Cloudflare's HTTP telemetry plus our application log.

## Why The Message Was Empty

Cloudflare's log UI has a primary `message` column.

The platform request log has a message:

```txt
POST https://.../api/realtimekit/webhook
```

Our app log is structured JSON and does not currently include a top-level
`message` field. Cloudflare still stores the fields, but the row can appear
with an empty message in the list view.

This can be improved by logging:

```json
{
  "message": "RealtimeKit webhook received",
  "event": "realtimekit_webhook_received",
  "status": "accepted",
  "eventType": "meeting.ended"
}
```

That would make the log easier to scan without changing behavior.

## Signature Verification Discovery

The real request includes:

```txt
dyte-signature: [redacted]
```

This means RealtimeKit/Dyte does provide a signature header in practice.

Current state:

- We receive the signature header.
- We do not yet verify it.
- We should not claim cryptographic verification until we confirm the official
  algorithm and signing secret source.

Next implementation step:

- Document the observed `dyte-signature` header in `docs/realtimekit-notes.md`.
- Search official Cloudflare/RealtimeKit/Dyte docs for the verification
  algorithm.
- If the algorithm is confirmed, implement verification.
- If not confirmed, keep signature verification as a V1 documented gap.

## Recommended Code Adjustments

The next webhook slice should update safe metadata extraction to recognize the
observed nested payload fields:

- `meeting.id` as `meetingId`
- `meeting.sessionId` as `sessionId`
- `meeting.endedAt` as `endedAt`
- top-level `reason`
- `dyte-webhook-id` as safe header metadata
- `dyte-uuid` as safe header metadata
- presence of `dyte-signature` as a boolean, without logging the signature value

The app should continue to avoid logging:

- the raw signature value
- auth tokens
- API tokens
- credentials
- unbounded payloads
